# Repository Guidelines

## Project Structure & Module Organization

MiniCPA is a strict TypeScript ESM CLI for managing CLIProxyAPI instances. The entry point is `src/cli.ts`. Keep command wiring in `src/commands/`, process lifecycle and health logic in `src/process/`, and download/update behavior in `src/update/`. Shared paths, state, config, HTTP, and filesystem helpers live directly under `src/`. Place unit tests beside their module as `*.test.ts`. `docs/` contains user-facing reference material; `dist/` is generated by the build and must not be edited manually. GitHub workflows are under `.github/workflows/`.

## Build, Test, and Development Commands

- `npm install` installs the Node.js 24+ development dependencies.
- `npm run dev -- --help` runs the TypeScript CLI without building it.
- `npm run typecheck` runs TypeScript validation without emitting files.
- `npm test` type-checks, then runs all `node:test` suites through `tsx`.
- `npm run build` emits the publishable CLI into `dist/`.
- `npm link` makes the local build available as the `cpa` command for manual smoke testing.
- `npm publish --dry-run --access public --provenance` validates the exact npm release flow without publishing.

## Coding Style & Naming Conventions

Use two-space indentation, double quotes, semicolons, and explicit TypeScript types where they clarify contracts. Preserve ESM imports with `.js` specifiers. Use `camelCase` for functions and variables, `PascalCase` for types/classes, and lowercase kebab-case for multiword filenames such as `lifecycle-cmd.ts`. No formatter or linter is configured; match surrounding code and keep changes focused.

## Testing Guidelines

Use the built-in `node:test` API with `node:assert/strict`. Name tests after the module they cover, for example `src/update/panel.test.ts`. Keep tests isolated with temporary directories and clean them in `afterEach`; do not depend on live GitHub releases, tokens, or a locally running CPA. Add regression coverage for lifecycle, update, locking, or integrity bugs before fixing them.

## Commit & Pull Request Guidelines

Follow the concise imperative style already used in history: `release: stabilize 0.0.5`, `docs: add MIT license`, or `Harden lifecycle checks`. Keep one concern per commit. Pull requests should state the behavior change, list commands run (usually `npm test` and `npm run build`), and call out any changes to update safety, configuration, or CLI output. Include CLI output only when it helps reviewers verify a user-visible change.

## Security & Configuration

Never commit `.env` files, npm tokens, GitHub tokens, or real CPA credentials. Preserve checksum verification by default, keep `--insecure` exceptional, and ensure MiniCPA credentials are not passed to CPA child processes.
